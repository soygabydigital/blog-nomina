SOMOS EQUIPO NÓMINA

Aprende de normalización de base de datos, Formas Normales, Ejemplo..

      Las formas normales son reglas que se aplican en el diseño de bases de datos relacionales para garantizar que los datos estén organizados de manera eficiente y se eliminen las redundancias innecesarias. A continuación se presentan las reglas de las tres formas normales principales:

FORMAS NORMALES

  • Primera Forma Normal (1FN:) cada atributo o campo de una tabla debe contener un solo valor y no una lista de valores separados por comas u otro separador. Además, cada fila o registro debe tener una clave única que lo identifique.

  • Segunda Forma Normal (2FN): una tabla está en 2FN si está en 1FN y todos sus atributos no clave dependen completamente de la clave primaria. En otras palabras, no debe haber dependencias parciales. Si un atributo no clave depende solo de una parte de la clave primaria, se debe separar en otra tabla.

  • Tercera Forma Normal (3FN): una tabla está en 3FN si está en 2FN y todos sus atributos no clave no dependen de otros atributos no clave. En otras palabras, no debe haber dependencias transitivas. Si un atributo no clave depende de otro atributo no clave, se debe separar en otra tabla.

      Además, hay otras formas normales, como la cuarta forma normal (4FN), la forma normal de Boyce-Codd (BCNF) y la forma normal de dominio-key (DKNF), que se utilizan para garantizar una mayor normalización y evitar problemas de redundancia y anormalidad de datos en la base de datos. Cada una de estas formas normales tiene sus propias reglas y requisitos específicos que se deben cumplir para garantizar una base de datos bien diseñada y eficiente.       Además de las tres formas normales principales (1FN, 2FN y 3FN) hay otras formas normales que se utilizan en el diseño de bases de datos para garantizar la integridad y la eficiencia de los datos. A continuación se describen brevemente estas formas normales adicionales:
  • Cuarta Forma Normal (4FN): Una tabla está en 4FN si no tiene múltiples valores multivaluados (atributos que pueden tener múltiples valores) y no tiene dependencias de varias claves (relaciones de muchos a muchos). La 4NF se enfoca en reducir redundancias y consiste en dividir una tabla en dos o más tablas, de tal manera que los datos que se relacionan estén en la misma tabla y las tablas que no tienen una relación directa se separen.
  • Quinta Forma Normal (5FN): Una tabla está en 5FN si está en 4FN y no hay pérdida de información cuando se descompone en dos o más tablas. La 5NF se enfoca en eliminar redundancias adicionales y consiste en descomponer una tabla en otras más pequeñas para reducir la duplicación de datos.
  • Sexta Forma Normal (6NF): Esta forma normal se enfoca en la representación de hechos temporales, es decir, la historia de una base de datos a lo largo del tiempo. La 6NF requiere que cada hecho sea registrado de manera independiente, incluyendo el momento en que se produjo, y que se utilicen claves temporales para representar la información histórica.

      Cada forma normal tiene sus propias reglas y requisitos específicos que deben cumplirse para garantizar una base de datos bien diseñada y eficiente. Es importante tener en cuenta que, aunque la normalización de una base de datos puede mejorar su eficiencia y consistencia, también puede aumentar su complejidad y dificultad de manejo. Por lo tanto, el diseño de la base de datos debe equilibrar la normalización con la facilidad de uso y la eficiencia.

EJEMPLO PRÁCTICO

    Supongamos que estamos diseñando una base de datos para una empresa que tiene un sistema de nómina y un sistema de presupuesto. Además, la empresa maneja información sobre sus terceros, como proveedores y clientes, y tiene un módulo de configuración para definir parámetros de la aplicación. Podemos modelar esta base de datos utilizando varias tablas.

  • Primera Forma Normal (1FN)
    Para cumplir con la 1FN, cada atributo debe contener un solo valor y cada registro debe tener una clave única. Podemos crear la siguiente tabla para almacenar la información de los empleados de la empresa:

    Tabla Empleados

    IDEmpleado (Clave primaria) Nombre Apellido Puesto Salario

    Aquí, el IDEmpleado es la clave primaria que identifica de forma única a cada empleado. La tabla cumple con la 1FN ya que cada atributo contiene un solo valor y cada registro tiene una clave única.

  • Segunda Forma Normal (2FN)
    Para cumplir con la 2FN, cada atributo que no sea clave debe depender completamente de la clave primaria. Podemos crear otra tabla para almacenar la información de los departamentos de la empresa:

    Tabla Departamentos

    IDDepartamento (Clave primaria) Nombre Descripcion

    Aquí, el IDDepartamento es la clave primaria que identifica de forma única a cada departamento. La tabla cumple con la 2FN ya que los atributos no clave (Nombre y Descripcion) dependen completamente de la clave primaria (IDDepartamento).

  • Tercera Forma Normal (3FN)
    Para cumplir con la 3FN, no debe haber dependencias transitivas entre los atributos no clave. Podemos crear una tabla para almacenar la información de los salarios de los empleados en un periodo determinado:

    Tabla Salarios

    IDSalario (Clave primaria) IDEmpleado (Clave foránea de la tabla Empleados) IDSemana (Clave foránea de la tabla Semanas) Monto

    Aquí, el IDSalario es la clave primaria que identifica de forma única a cada registro de salario. El IDEmpleado y el IDSemana son claves foráneas que relacionan la tabla de salarios con las tablas de empleados y semanas, respectivamente. La tabla cumple con la 3FN ya que no hay dependencias transitivas entre los atributos no clave (IDEmpleado, IDSemana y Monto).

  • Cuarta Forma Normal (4FN)
    Para cumplir con la 4FN, no debe haber múltiples valores multivaluados en una tabla. Podemos crear una tabla para almacenar la información de los departamentos a los que pertenece cada empleado:

    Tabla EmpleadosDepartamentos

    IDEmpleado (Clave foránea de la tabla Empleados) IDDepartamento (Clave foránea de la tabla Departamentos)

    Aquí, la tabla EmpleadosDepartamentos tiene dos claves foráneas, que relacionan la tabla de empleados con la tabla de departamentos. La tabla cumple con la 4FN ya que no tiene múltiples valores multivaluados.

  • Quinta Forma Normal (5FN)
    Para cumplir con la 5FN, la tabla debe estar en la 4FN y no debe haber pérdida de información cuando se descompone en dos o más tablas. Podemos crear una tabla para almacenar la información de los presupuestos de cada departamento:

    Tabla PresupuestosDepartamentos

    IDPresupuesto (Clave primaria) IDDepartamento (Clave foránea de la tabla Departamentos) Año Monto

    Aquí, el IDPresupuesto es la clave primaria que identifica de forma única a cada presupuesto, y el IDDepartamento es una clave foránea que relaciona la tabla de presupuestos con la tabla de departamentos. La tabla cumple con la 5FN ya que está en la 4FN y no hay pérdida de información al descomponerla en dos o más tablas.

  • Sexta Forma Normal (6FN)
    Para cumplir con la 6FN, la tabla debe estar en la 5FN y no debe haber dependencias funcionales entre las claves. Podemos crear una tabla para almacenar la información de los terceros que interactúan con la empresa:

    Tabla Terceros

    IDTercero (Clave primaria) TipoTercero Nombre Direccion Telefono

    Aquí, el IDTercero es la clave primaria que identifica de forma única a cada tercero, y el TipoTercero es un atributo que indica si se trata de un proveedor o un cliente. La tabla cumple con la 6FN ya que está en la 5FN y no hay dependencias funcionales entre las claves.

      En resumen, siguiendo las reglas de las formas normales podemos diseñar una base de datos eficiente y bien estructurada, que garantice la integridad de la información y facilite la manipulación de los datos de una empresa o cualquier otro tipo de organización.
