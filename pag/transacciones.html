SOMOS EQUIPO NÓMINA

Aprende de relaciones de base de datos, Nomenclatura de relación,, parámetros y normas.

Una transacción de base de datos es un área única de la base de datos donde se llevan a cabo múltiples operaciones de datos y se escriben como un todo.

Estas operaciones pueden ser operaciones de creación, lectura, actualización o eliminación.

Durante el proceso de una transacción, la base de datos se encuentra en un estado inconsistente porque hay operaciones en curso que están realizando cambios en la base de datos. La base de datos vuelve a un estado más coherente cuando se han confirmado las operaciones.

Para que una transacción sea exitosa, significa que todas las operaciones realizadas han sido comprometidas.

Las transacciones de la base de datos son muy importantes para garantizar la coherencia de su base de datos cuando se realizan varias operaciones al mismo tiempo. También le brinda una forma de recuperar los cambios que pueden haber ocurrido debido a la falla o mal uso accidental de una operación.

PROPIEDADES ACID DE LAS TRANSACCIONES

      ACID es un acrónimo que significa Atomicidad, Consistencia, Aislamiento y Durabilidad. Repasemos cada término para comprender cómo se relacionan con las transacciones.

  • Atomicidad: La atomicidad en una transacción de base de datos significa que todos los cambios realizados durante esa transacción se tratan como un "paquete" de cambios. Esto significa que cuando intenta modificar su base de datos, todos los cambios ocurren al mismo tiempo o ninguno de ellos ocurre en absoluto.

  • Consistencia: La coherencia en las bases de datos significa que los datos almacenados en la base de datos están siempre en un estado válido y coherente. Por ejemplo, si la base de datos contiene restricciones, como claves primarias, claves externas, etc., siempre debe cumplir con las reglas que rodean la restricción.

  • Aislamiento: La capacidad de múltiples transacciones para ejecutarse sin interferir entre sí se conoce como Aislamiento. El nivel de aislamiento de una transacción determina cómo los cambios realizados por esa transacción son visibles para otras transacciones.

  • Durabilidad: La durabilidad garantiza que sus datos permanezcan seguros, incluso en caso de circunstancias imprevistas. Cuando se confirma una transacción, sus cambios deben permanecer en la base de datos, incluso si experimenta un mal funcionamiento o un corte de energía.

¿CÓMO AFECTA NO UTILIZAR CORRECTAMENTE ACID?

  • Dirty reads (Lecturas sucias): Es el problema más importante de todos. Supone que las transacciones en curso puedan leer el resultado de otras transacciones aún no confirmadas. Por ejemplo, vamos a suponer que tenemos dos transacciones activas (A y B). Inicialmente la transacción A lee un valor X de una tabla que, por ejemplo, es 0. Durante dicha transacción el valor de X se cambia a 10, pero aún la transacción no se ha cometido, por lo que en la tabla X = 0. Ahora la transacción B accede al valor X y obtiene ¡¡X = 10!! un valor que está usando A y que aún no se ha cometido. Supongamos que ahora se anula la transacción A. El resultado sería X = 0 en la tabla y X = 10 en la transacción B por lo que hemos llegado a un estado muy grave de inconsistencia.

  • Non-Repeatable reads (Lecturas no repetibles): Ocurre cuando una transacción activa vuelve a leer un dato cuyo valor difiere con respecto al de la anterior lectura. Lo vemos más claro con un ejemplo. Supongamos que una transacción activa, A, lee un valor X = 0. En este momento otra transacción B modifica el valor de X, por ejemplo X = 10, y se comete dicha transacción. Si ahora durante la transacción A se vuelve a leer el valor X obtendríamos 10 en lugar del 0 que se esperaba. Aunque a primera vista este problema no parezca muy importante en realidad sí que lo es, sobre todo cuando X es una clave primaria o ajena. En este caso se puede originar una gran pérdida de consistencia en nuestra base de datos.

  • Phantom reads (Lecturas fantasma): Este supone el menor problema que se nos puede plantear con respecto a las transacciones. Sucede cuando una transacción en un momento lanza una consulta de selección con una condición y recibe en ese momento N filas y posteriormente vuelve a lanzar la misma consulta junto con la misma condición y recibe M filas con M > N. Esto es debido a que durante el intervalo que va de la primera a la segunda lectura se insertaron nuevas filas que cumplen la condición impuesta en la consulta.

¿CÓMO APLICAR LAS PROPIEDADES DE LAS TRANSACCIONES ACID?

  • //inicia la transacción
  • START TRANSACTION.
  • //efectuamos nuestra operación
  • UPDATE empleados SET nombre=”Jose” WHERE empleado=1;
  • //realizamos los cambios
  • COMMIT;

    Si falla el registro entonces aplicamos el ROLLBACK para revertir el registro que estaba anteriormente registrado satisfactoriamente.

  • //inicia la transacción
  • START TRANSACTION.
  • //efectuamos nuestra operación
  • UPDATE empleados SET nombre=”Jose” WHERE empleado=1;
  • //realizamos los cambios
  • ROLLBACK;

    Para aplicar la consistencia de los datos, podemos utilizar un motor de almacenamiento como los es InnoDB para que respete la integridad de las relaciones.

  • CREATE TABLE customers (a INT, b CHAR (20), INDEX (a)) ENGINE=InnoDB;

NIVELES DE AISLAMIENTO

  • Read uncommitted (Lectura sin confirmación): En la práctica casi no se suele utilizar este nivel de aislamiento ya que es propenso a sufrir todos los problemas anteriormente descritos. En este nivel una transacción puede ver los resultados de transacciones aún no cometidas. Podemos apreciar que en este nivel no existe aislamiento alguno entre transacciones.

  • Read committed (Lectura confirmada): Es el predeterminado para la mayoría de gestores de bases de datos relacionales. Supone que dentro de una transacción únicamente se pueden ver los cambios de las transacciones ya cometidas. Soluciona el problema de las lecturas sucias, pero no el de las lecturas no repetibles ni tampoco el de las lecturas fantasmas.

  • Repeatable read (Lectura repetible): Define que cualquier tupla leída durante el transcurso de una transacción es bloqueada. De esta forma se soluciona, además de las lecturas sucias, el problema de las lecturas no repetibles. Aunque en dicho nivel se siguen dando las lecturas fantasmas.

  • Serializable (Lecturas en serie): Soluciona todos los problemas descritos. Para ello ordena las transacciones con el objetivo de que no entren en conflicto. Este nivel de aislamiento es bastante problemático ya que es, con diferencia, el que más sacrifica en rendimiento y concurrencia.

    Si deseamos conocer el nivel de aislamiento con el que actualmente está trabajando nuestro servidor no tenemos más que lanzar la sentencia:

  • SHOW VARIABLES LIKE 'tx_isolation'.

    Para cambiarlo directamente desde la línea de comando de MySQL lo podemos hacer con la sentencia:

  • SET {GLOBAL | SESSION} TRANSACTION ISOLATION LEVEL {READ UNCOMMITTED | READ COMMITED | REPEATABLE READ | SERIALIZABLE}

EJEMPLO DE TRANSACCIÓN USANDO MYSQL
(CONCURRENCIA)

    Supongamos que tenemos una actualización de datos de la tabla auto de la base de datos:

    Las sentencias SQL son atómicas. Es decir, si ejecutas algo como esto:

  • UPDATE Cars SET Sold = Sold + 1

    Nadie puede cambiar la Sold variable durante esta declaración. Siempre se incrementa en 1, incluso si alguien más está ejecutando la misma declaración al mismo tiempo.

    El problema ocurre si tiene sentencias que dependen unas de otras:

  • a = SELECT Sold FROM Cars;
  • UPDATE Cars SET Sold = a + 1;

    Entre estas consultas, otro usuario puede cambiar la tabla Autos y actualizar Vendidos. Para evitar esto, envuélvelo en una transacción:

  • BEGIN;
  • a = SELECT Sold FROM Cars;
  • UPDATE Cars SET Sold = a + 1;
  • COMMIT;

    Las transacciones son compatibles con InnoDB, pero no con MyISAM.

    Incluso en una transacción no hay garantía. MySQL, de forma predeterminada, realiza el bloqueo de nivel de fila/bloqueo de tabla solo para operaciones de escritura, incluso en transacciones. Para que tenga un valor correcto, es necesario poner FOR UPDATE en SELECT, o especificar un nivel de aislamiento más alto en TRANSACTION como SERIALIZABLE.

CONFIGURAR DE MANERA ÓPTIMA EL ARCHIVO DE CONFIGURACIÓN DE POSTGRESQL
(CONCURRENCIA)

    El comportamiento de PostgreSQL en nuestro sistema se puede controlar con tres ficheros de configuración que se encuentran en el directorio de datos donde inicializamos nuestro cluster PostgreSQL (En nuestro caso /var/pgsql/data). Estos tres ficheros son:

  • pg_hba.conf: Este fichero se utiliza para definir los diferentes tipos de accesos que un usuario tiene en el cluster.
  • pg_ident.conf: Este fichero se utiliza para definir la información necesaria en el caso que utilicemos un acceso del tipo ident en pg_hba.conf .
  • postgresql.conf: En este fichero podemos cambiar todos los parámetros de configuración que afectan al funcionamiento y al comportamiento de PostgreSQL en nuestra máquina.
